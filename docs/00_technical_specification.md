# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## Проект: Платформа краткосрочной аренды недвижимости

---

# 1. Назначение системы

Спроектировать и реализовать реляционную базу данных для платформы краткосрочной аренды жилой недвижимости.

Система предназначена для:

- размещения объектов недвижимости владельцами;
- поиска и бронирования объектов гостями;
- управления календарём доступности;
- хранения истории цен;
- фиксации бронирований и платежей;
- хранения отзывов;
- реализации ролевой модели доступа (RBAC);
- получения аналитических данных.

Проект должен быть реализован с учетом требований к целостности данных, нормализации, ограничений и масштабируемости.

---

# 2. Границы проекта

В рамках проекта реализуется:

- схема базы данных;
- описание таблиц и связей;
- ограничения целостности;
- ролевая модель доступа;
- перечень бизнес-инвариантов;
- перечень бизнес-задач.

Не требуется:

- реализация API;
- реализация пользовательского интерфейса;
- интеграция с внешними сервисами.

---

# 3. Основные доменные области

Система должна поддерживать следующие области данных:

1. Пользователи
2. Роли и разрешения (RBAC)
3. Объекты недвижимости
4. Календарь доступности
5. Ценообразование
6. Бронирования
7. Платежи
8. Отзывы
9. Фото объектов недвижимости
10. Аналитика (на уровне запросов)

---

# 4. Функциональные требования

---

## 4.1 Пользователи

Система должна обеспечивать:

- хранение учетной записи пользователя;
- хранение контактных данных;
- хранение даты регистрации;
- хранение статуса учетной записи (активен, заблокирован и т.д.);
- возможность наличия нескольких ролей у одного пользователя.

Один пользователь может:

- быть гостем;
- быть владельцем;
- быть администратором;
- совмещать несколько ролей одновременно.

---

## 4.2 Ролевая модель (RBAC)

### 4.2.1 Общие требования

- Пользователь может иметь одну или несколько ролей.
- Каждая роль содержит набор разрешений.
- Разрешение - атомарное право на выполнение конкретного действия над ресурсом.
- Итоговые права пользователя формируются как объединение разрешений всех его ролей.

### 4.2.2 Предопределённые роли

Система должна поддерживать следующие системные роли:

- GUEST
- OWNER
- ADMIN

Архитектура должна позволять добавление новых ролей без изменения схемы.

### 4.2.3 Разрешения

Разрешения определяются как комбинация:

- ресурс
- действие

Ресурсы:

- USER
- LISTING
- PHOTO
- AVAILABILITY
- PRICE_RULE
- BOOKING
- PAYMENT
- REVIEW
- REPORT

Действия:

- CREATE
- READ
- UPDATE
- DELETE
- CANCEL
- REFUND
- MODERATE

### 4.2.4 Матрица разрешений

#### GUEST

- LISTING:READ
- PHOTO:READ
- BOOKING:CREATE
- BOOKING:READ (только свои)
- BOOKING:CANCEL (только свои, в допустимых статусах)
- PAYMENT:CREATE (по своему бронированию)
- PAYMENT:READ (только свои)
- REVIEW:CREATE (по завершённому бронированию)
- REVIEW:READ

#### OWNER

- LISTING:CREATE
- LISTING:READ (свои)
- LISTING:UPDATE (свои, включая смену статуса публикации)
- PHOTO:CREATE/READ/DELETE (для своих объектов)
- AVAILABILITY:CREATE/UPDATE (для своих объектов)
- PRICE_RULE:CREATE/UPDATE/DELETE (для своих объектов)
- BOOKING:READ (по своим объектам)
- BOOKING:UPDATE (подтверждение/отклонение)
- BOOKING:CANCEL (для своих объектов, с обязательным указанием причины при отмене подтвержденной брони)
- REPORT:READ (по своим объектам)

#### ADMIN

- USER:READ/UPDATE
- LISTING:READ/UPDATE (любые, только модерация и смена статусов)
- PHOTO:READ/DELETE (любые, только модерация)
- BOOKING:READ/UPDATE/CANCEL (любые)
- PAYMENT:READ/REFUND (любые)
- REVIEW:READ/MODERATE/DELETE
- REPORT:READ (глобальная аналитика)

---

# 5. Объекты недвижимости (LISTING)

Система должна хранить:

- владельца объекта;
- тип объекта;
- адрес (страна, город, улица);
- вместимость;
- количество комнат;
- описание;
- статус публикации (активен/скрыт/заблокирован);
- дату создания;
- дату обновления.
- фотографии объекта (до 20 фото на одно объявление).

Один владелец может иметь несколько объектов.

---

# 6. Календарь доступности (AVAILABILITY)

Для каждого объекта система должна:

- хранить доступность по дням: одна дата = одна запись;
- хранить статусы доступных дат;
- позволять владельцу добавлять доступные даты и менять им статус;
- не удалять созданные даты доступности, а только переводить между статусами;
- учитывать активные бронирования (в том числе ожидающие оплаты) при смене статусов;
- запрещать перевод даты в недоступный/удаляемый для бронирования статус, если дата удерживается активной бронью.

Рекомендуемая логика статусов даты доступности:

- `AVAILABLE` - дата доступна для бронирования;
- `HELD` - дата временно удерживается активной бронью до оплаты;
- `BOOKED` - дата занята подтвержденной бронью;
- `BLOCKED` - дата вручную закрыта владельцем.

---

# 7. Ценообразование (PRICE_RULE)

Система должна:

- хранить базовую цену за ночь;
- хранить специальные цены на период;
- хранить историю изменения цен;
- поддерживать разные периоды действия цены.

Цена бронирования фиксируется в момент создания и не изменяется задним числом.

---

# 8. Бронирования (BOOKING)

Бронирование должно содержать:

- информацию о госте;
- информацию об объекте размещения;
- выбранные даты проживания;
- количество гостей;
- итоговую стоимость;
- статус бронирования.

Бронирование может включать как непрерывный, так и разрывный набор дат.
Все даты внутри одного бронирования должны относиться только к одному объекту размещения.

Статусы бронирования:

- CREATED
- PAYMENT_PENDING
- CONFIRMED
- EXPIRED
- CANCELLED
- COMPLETED

Система должна:

- запрещать пересечение активных бронирований по дням доступности;
- запрещать создание бронирования без выбранных дат проживания (в бронировании должен быть минимум один день доступности объекта);
- хранить дату создания;
- хранить дату изменения статуса;
- хранить время истечения бронирования до начала оплаты (`booking_expires_at`);
- разрешать старт платежной сессии только в течение 5 минут после создания бронирования;
- при создании бронирования переводить выбранные дни объекта в статус `HELD`;
- переводить бронирование в `PAYMENT_PENDING` при успешном создании платежной сессии;
- запрещать отмену бронирования в статусе `PAYMENT_PENDING`;
- автоматически подтверждать бронирование при успешной оплате и переводить связанные дни в статус `BOOKED`;
- переводить бронирование в `EXPIRED`, если срок брони или платежной сессии истек, и снимать удержание дней обратно в `AVAILABLE`;
- при отмене до оплаты снимать удержание дней обратно в `AVAILABLE`;
- разрешать отмену подтвержденного бронирования владельцем только с обязательным указанием причины;
- обеспечивать невозможность повторного отзыва по одному бронированию.

---

# 9. Платежи (PAYMENT)

Система должна:

- хранить связь платежа с бронированием;
- хранить сумму;
- хранить валюту;
- хранить статус платежа (INITIATED, PAID, FAILED, CANCELLED, EXPIRED, PARTIALLY_REFUNDED, REFUNDED);
- хранить дату операции.
- хранить идентификатор платежной сессии внешнего провайдера;
- хранить время истечения платежной сессии, полученное от провайдера;
- разрешать только одну платежную сессию на одно бронирование;
- запрещать запуск платежной сессии после истечения `booking_expires_at`.
- поддерживать полный и частичный возврат по платежу;
- при отмене подтвержденного бронирования владельцем переводить платеж в статус возврата (`REFUNDED` или промежуточный статус возврата с последующим `REFUNDED`).

---

# 10. Отзывы (REVIEW)

Система должна:

- позволять гостю оставить отзыв только по завершённому бронированию;
- хранить числовую оценку;
- хранить текст отзыва;
- хранить дату создания;
- обеспечивать один отзыв на одно бронирование;
- поддерживать модерацию администратором.

---

# 11. Бизнес-инварианты

База данных должна обеспечивать:

1. Невозможность двойного активного бронирования одного и того же дня доступности объекта.
2. Невозможность создания отзыва без завершённого бронирования.
3. Невозможность более одного отзыва на одно бронирование.
4. Невозможность бронирования деактивированного объекта.
5. Невозможность подтверждения бронирования при отсутствии доступности дат или при истекшем сроке оплаты.
6. Невозможность выполнения операций без соответствующего разрешения роли.
7. Ссылочную целостность всех связей.
8. Невозможность запуска более одной платежной сессии для одного бронирования.
9. Невозможность запуска платежной сессии после истечения `booking_expires_at`.
10. Невозможность отмены бронирования в статусе `PAYMENT_PENDING`.
11. Невозможность создания более 20 фотографий для одного объявления.
12. Невозможность бронирования дат вне явно заданных владельцем периодов доступности.
13. Невозможность физического удаления созданных дат доступности (разрешена только смена статуса).
14. Невозможность смены статуса даты доступности, если дата удерживается активной бронью.
15. Невозможность отмены подтвержденного бронирования владельцем без причины.
16. Невозможность физического удаления объявлений (допускается только смена статуса публикации).
17. Невозможность существования бронирования без хотя бы одного выбранного дня доступности объекта.

---

# 12. Нефункциональные требования

1. Схема должна соответствовать не ниже 3НФ.
2. Должны быть реализованы:
    - первичные ключи;
    - внешние ключи;
    - уникальные ограничения;
    - CHECK-ограничения;
    - индексы.
3. Должна быть обеспечена масштабируемость.
4. Должна быть предусмотрена возможность расширения ролей и разрешений.

---

# 13. Примеры бизнес-задач

База данных должна позволять:

1. Найти доступные объекты в заданном диапазоне дат.
2. Рассчитать доход владельца за период.
3. Определить среднюю загрузку объекта.
4. Найти объекты с самым высоким рейтингом.
5. Получить историю изменения цен по объекту.
6. Найти пользователей с наибольшим количеством бронирований.
7. Рассчитать среднюю длительность проживания.
8. Получить глобальную выручку платформы за период.
9. Найти отмененные бронирования и частоту отмен.
10. Определить наиболее популярные города.
