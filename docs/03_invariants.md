# Бизнес-инварианты и их обеспечение на уровне БД

Документ фиксирует инварианты из ТЗ и способ их реализации в текущей схеме.

## 1. Двойное активное бронирование одного дня невозможно
- Смысл: один и тот же день доступности не может одновременно участвовать в двух активных бронированиях.
- Реализация:
  - триггер `trg_booking_days_prevent_active_overlap` в `V006`
  - проверка конфликтов по статусам брони `created`, `payment_pending`, `confirmed`
  - блокировка строки дня через `FOR UPDATE` для защиты от конкурентных гонок.

## 2. Отзыв только по завершенному бронированию
- Смысл: нельзя оставить отзыв, если бронь не перешла в `completed`.
- Реализация:
  - триггер `trg_reviews_require_completed_booking` в `V008`
  - проверка `bookings.status = completed` перед `INSERT/UPDATE` в `reviews`
  - чтение строки брони с `FOR UPDATE`.

## 3. Один отзыв на одно бронирование
- Смысл: дублирование отзывов по одной брони запрещено.
- Реализация:
  - `UNIQUE (booking_id)` в таблице `reviews` (`V008`).

## 4. Нельзя бронировать неактивное объявление
- Смысл: привязка дней к брони допустима только для `listings.status = active`.
- Реализация:
  - триггер `trg_booking_days_prevent_active_overlap` в `V006`
  - явная проверка статуса объявления перед вставкой/обновлением `booking_days`.

## 5. Нельзя подтверждать/удерживать бронь без валидной доступности
- Смысл: операции с активной бронью допустимы только по существующим дням выбранного объекта.
- Реализация:
  - `booking_days` содержит FK:
    - `(booking_id, listing_id) -> bookings(id, listing_id)`
    - `(listing_id, availability_day_id) -> listing_availability_days(listing_id, id)`
  - триггер `trg_booking_days_prevent_active_overlap` проверяет наличие дня и отсутствие активного конфликта.

## 6. Операции RBAC выполняются только при наличии прав
- Смысл: права заданы через роли и разрешения.
- Реализация в схеме данных:
  - `roles`, `permissions`, `role_permissions`, `user_roles` (`V002`, `V003`)
  - консистентность обеспечивается PK/FK/UNIQUE;
  - прикладной слой интерпретирует права для конкретных бизнес-операций.

## 7. Ссылочная целостность всех ключевых связей
- Смысл: ссылки на связанные сущности не могут указывать на несуществующие записи.
- Реализация:
  - FK по всем основным зависимостям (`users`, `listings`, `booking_days`, `payments`, `reviews` и др.).

## 8. Одна платежная сессия на одну бронь
- Смысл: нельзя создать вторую платежную сессию для того же бронирования.
- Реализация:
  - `UNIQUE (booking_id)` в `payments` (`V007`).

## 9. Нельзя стартовать платеж после `booking_expires_at`
- Смысл: сессия оплаты создается только пока окно оплаты брони не истекло.
- Реализация:
  - триггер `trg_payments_require_not_expired_booking` в `V007`
  - сравнение `now()` с `bookings.booking_expires_at`
  - блокировка строки брони `FOR UPDATE`.

## 10. Нельзя отменить бронь в `payment_pending`
- Смысл: в процессе платежной сессии отмена запрещена.
- Реализация:
  - триггер `trg_bookings_validate_status_transition` в `V006`
  - запрет перехода `payment_pending -> cancelled`.

## 11. Не более 20 фото на объявление
- Смысл: ограничение фотогалереи одного объявления.
- Реализация:
  - `slot SMALLINT` с `CHECK (slot BETWEEN 1 AND 20)` в `listing_photos` (`V004`)
  - `UNIQUE (listing_id, slot)` исключает два фото в одном слоте
  - фактически максимум 20 слотов = максимум 20 фото.

## 12. Нельзя бронировать дни вне явной доступности владельца
- Смысл: в бронь попадают только существующие записи `listing_availability_days`.
- Реализация:
  - FK `booking_days` к `listing_availability_days`
  - триггер `trg_booking_days_prevent_active_overlap` проверяет существование и принадлежность дня объекту.

## 13. Нельзя физически удалить дату доступности
- Смысл: даты живут как история; меняется только статус.
- Реализация:
  - триггер `trg_listing_availability_days_prevent_delete` в `V005`.

## 14. Нельзя менять статус даты, удерживаемой активной бронью
- Смысл: защищаем удержание/подтверждение дней от ручной смены статуса.
- Реализация:
  - триггер `trg_listing_availability_days_block_status_change_on_active_hold` в `V006`.

## 15. Отмена подтвержденной брони владельцем только с причиной
- Смысл: отмена без причины недопустима.
- Реализация:
  - `CHECK` `chk_bookings_cancel_reason_required` в `bookings` (`V006`).

## 16. Нельзя физически удалить объявление
- Смысл: объявления не удаляются, только меняют `status`.
- Реализация:
  - триггер `trg_listings_prevent_delete` в `V004`.

## 17. Бронь не может существовать без выбранных дней
- Смысл: у бронирования должен быть минимум один день проживания.
- Реализация:
  - триггерная проверка в `V006` (контроль через таблицу `booking_days`),
  - плюс требования прикладного слоя к транзакционному сценарию создания брони.

## 18. Дополнительные инварианты целостности цен и платежей

### 18.1 Цены и валюты
- `base_prices.amount_in_minor > 0`
- `listing_availability_days.override_*` заполняются парой (оба поля вместе)
- `price_history` хранит только осмысленные изменения (не допускает полностью пустые «изменения»)
- согласованность `price_history` с днем доступности обеспечена составным FK.

### 18.2 Платежи и возвраты
- `payments.amount_in_minor > 0`
- `payments.refunded_amount_in_minor` в диапазоне `[0; amount_in_minor]`
- `provider_payment_session_expires_at > initiated_date`
- `provider_payment_session_id` уникален (идемпотентность по внешнему провайдеру).

## 19. Примечание по распределению ответственности

Инварианты, критичные для целостности при конкурентной нагрузке, закреплены на уровне БД (PK/FK/UNIQUE/CHECK/trigger + row locking).
Прикладной слой отвечает за оркестрацию бизнес-процесса (сценарии переходов статусов, интеграция с платежным провайдером, RBAC-проверки на уровне API).
