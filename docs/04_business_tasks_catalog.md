# Каталог бизнес-задач

Документ фиксирует операционные и аналитические задачи, которые должна поддерживать БД.
SQL-реализация задач хранится отдельно и не приводится в этом каталоге.

## 1. Операционные задачи

### 1.1 Поиск доступных объектов на набор дат
- Цель: показать гостю объявления, которые доступны на выбранные даты.
- Вход: набор дат, локация/фильтры, число гостей.
- Выход: список объявлений с актуальной стоимостью и статусом доступности.

### 1.2 Создание бронирования
- Цель: зафиксировать выбранные дни за пользователем и создать бронь.
- Вход: `listing`, набор дней, число гостей.
- Выход: запись бронирования со статусом `created`, рассчитанная итоговая сумма, срок `booking_expires_at`.

### 1.3 Запуск платежной сессии
- Цель: начать оплату по созданной брони в разрешенное окно.
- Вход: `booking_id`, параметры платежного провайдера.
- Выход: запись платежа со статусом `initiated`, сохраненный `provider_payment_session_id` и срок истечения сессии.

### 1.4 Подтверждение оплаты и перевод брони в confirmed
- Цель: завершить сценарий успешной оплаты.
- Вход: событие от провайдера/результат оплаты.
- Выход: обновленный статус платежа (`paid`), бронирование `confirmed`, дни переведены в `booked`.

### 1.5 Обработка истечения брони и/или платежной сессии
- Цель: освободить удержанные дни, если оплата не завершена вовремя.
- Вход: текущее время и статусы бронирования/платежа.
- Выход: бронирование `expired`, дни возвращены в `available`, платеж отмечен как `expired`/`cancelled` при необходимости.

### 1.6 Отмена бронирования
- Цель: корректно отменить бронь с учетом статуса и роли.
- Вход: `booking_id`, инициатор отмены, причина (если обязательна).
- Выход: статус `cancelled`, корректная обработка дней доступности и связанных платежей.

### 1.7 Полный или частичный возврат платежа
- Цель: отразить возврат средств при отмене/споре.
- Вход: `payment_id`, сумма возврата.
- Выход: обновление `refunded_amount_in_minor`, статуса (`partially_refunded` или `refunded`).

### 1.8 Управление календарем доступности владельцем
- Цель: владелец задает доступные даты и меняет их статусы.
- Вход: `listing_id`, набор дат, целевые статусы.
- Выход: актуальные записи `listing_availability_days` без физического удаления дат.

### 1.9 Управление ценами владельцем
- Цель: обновлять базовую цену и дневные override-цены.
- Вход: `listing_id`, валюта, сумма, период/даты.
- Выход: актуальная цена и запись в `price_history`.

### 1.10 Модерация объявлений, фото и отзывов администратором
- Цель: реагировать на нарушения правил сервиса.
- Вход: объект модерации, решение администратора.
- Выход: измененные статусы объявлений, удаленные/скрытые фото, модерированные отзывы.

### 1.11 Публикация и обновление фото объявления
- Цель: поддерживать фотогалерею объявления.
- Вход: `listing_id`, фото, слот.
- Выход: привязка фото к объявлению с соблюдением лимита до 20 фото.

### 1.12 Создание отзыва после проживания
- Цель: собрать обратную связь гостя по завершенной брони.
- Вход: `booking_id`, оценка, текст.
- Выход: один отзыв на бронь, доступный для чтения и модерации.

## 2. Аналитические задачи

### 2.1 Доход владельца за период
- Цель: считать выручку владельца по подтвержденным/завершенным бронированиям.
- Вход: `owner_id`, период.
- Выход: сумма выручки и агрегаты по объявлениям.

### 2.2 Глобальная выручка платформы
- Цель: оценить общий оборот платформы за интервал.
- Вход: период.
- Выход: общая выручка, возвраты, чистая выручка.

### 2.3 Средняя загрузка объекта
- Цель: измерить использование календаря доступности конкретного объявления.
- Вход: `listing_id`, период.
- Выход: доля занятых дней относительно доступных.

### 2.4 Рейтинг объектов
- Цель: ранжировать объекты по качеству опыта гостей.
- Вход: период/регион/доп. фильтры.
- Выход: средний рейтинг, количество отзывов, топ объектов.

### 2.5 История изменения цен
- Цель: анализ динамики базовой и дневной цены.
- Вход: `listing_id`, период.
- Выход: лента изменений из `price_history` с источником изменения.

### 2.6 Самые активные пользователи
- Цель: определить пользователей с наибольшим количеством бронирований.
- Вход: период.
- Выход: рейтинг пользователей по числу броней и сумме оплат.

### 2.7 Средняя длительность проживания
- Цель: анализ поведения гостей по длине поездок.
- Вход: период/сегменты.
- Выход: среднее число дней на бронь, распределение по диапазонам.

### 2.8 Частота отмен
- Цель: мониторить стабильность бронирований.
- Вход: период, сегмент (город/объект/пользователь).
- Выход: доля отмененных бронирований, причины отмен.

### 2.9 Популярные города
- Цель: выявить города с максимальным спросом.
- Вход: период.
- Выход: рейтинг городов по числу бронирований, оплат и выручке.

### 2.10 Эффективность модерации
- Цель: оценить влияние админ-действий на качество каталога.
- Вход: период.
- Выход: количество модераций по типам сущностей, динамика блокировок и удалений контента.

## 3. Технические примечания к каталогу

- Каталог описывает только бизнес-задачи и ожидаемые результаты.
- Конкретные SQL-запросы и их оптимизация оформляются отдельно в `sql/operational/` и `sql/analytics/`.
- Проверка корректности запросов документируется в `artifacts/explain/` и `artifacts/snapshots/`.
