# Контекст проекта

## Назначение

`Rental Housing Platform DB` - учебный проект по проектированию реляционной базы данных для платформы краткосрочной аренды недвижимости.

Полная формулировка требований вынесена в отдельный документ:
- `docs/00_technical_specification.md`

## Цель

Спроектировать корректную и расширяемую схему БД, которая покрывает операционные сценарии платформы и поддерживает аналитические запросы.

## Границы проекта

Входит:

- схема БД и ее эволюция через миграции;
- ограничения целостности и бизнес-инварианты;
- ролевая модель доступа (RBAC) на уровне данных;
- SQL-запросы для операционных и аналитических задач;
- документация по модели данных.

Не входит:

- API;
- UI;
- интеграции с внешними сервисами.

## Доменные области

1. Пользователи.
2. Роли и разрешения (RBAC).
3. Объекты недвижимости.
4. Календарь доступности.
5. Ценообразование.
6. Бронирования.
7. Платежи.
8. Отзывы.
9. Фото объектов недвижимости.
10. Аналитические выборки.

## Ключевые требования к качеству модели

- Нормализация не ниже 3НФ.
- Использование PK, FK, UNIQUE, CHECK и индексов.
- Ссылочная целостность всех связей.
- Предотвращение логических аномалий (двойное бронирование, дубли отзывов и т.д.).
- Расширяемость модели ролей и разрешений.
- Бронирование блокирует слот сразу при создании и живет 5 минут до старта оплаты (`booking_expires_at`).
- Для бронирования допускается только одна платежная сессия; при старте сессии бронь переходит в `PAYMENT_PENDING`.
- В `PAYMENT_PENDING` отмена бронирования запрещена; успех оплаты автоматически переводит бронь в `CONFIRMED`.
- Время жизни платежной сессии берется от провайдера и сохраняется как `provider_expires_at + 10 секунд`.
- Доступность хранится по дням (`1 день = 1 запись`) со статусами (`AVAILABLE`, `HELD`, `BOOKED`, `BLOCKED`).
- Бронирование связывается с днями доступности как many-to-many через связующую таблицу (например, `booking_days`).
- Созданные даты доступности не удаляются физически; владелец может только менять их статус.
- Смена статуса даты запрещена, если дата удерживается активной бронью.
- Для подтвержденной брони отмена владельцем допускается только с обязательной причиной и запуском возврата.
- Для одного объявления допускается не более 20 фотографий.
